- 2026-07-21 — **Real per-zone irrigation `Schedule` (agronomic interval).** The long-deferred `Schedule` model, scoped to manual agronomic inputs (weather/soil feeds stay Phase 1.5+/Parking Lot). New `irrigation.schedules` table (one row per zone — unique no-FK `zone_id`, app-validated via `core_service.get_zone`; migration `ffab7af3eca1`): four inputs (soil available-water mm/m, root depth m, allowable depletion %, peak crop use mm/day) plus the **client-computed** `interval_days` + `readily_available_water_mm` stored **verbatim** — the interval comes from the already-tested `SchedulingCalculators.MaxIrrigationInterval` (RAW = AWC × root × MAD%; interval = RAW ÷ peak), keeping the module's no-server-math rule intact. `PUT`/`GET /api/v1/irrigation/zones/{id}/schedule` (upsert — the module's first update path; GET returns 200+null when unset). **Status rewire:** `core/service.py` gained `_schedule_interval_lookup` (same deferred-import inversion as `_last_run_lookup`); effective interval = floor(schedule interval) when a schedule exists, else the manual `zone.irrigation_interval_days` (unchanged fallback — `_compute_zone_status` itself untouched). Frontend: `IScheduleStore`/`HttpScheduleStore` in eager Core (ZoneOverview reads it; same DI rule as the design store), editor `ZoneSchedule.razor` in the lazy module (it calls the lazy calculators lib) with live recompute + next-due preview, and a real "Irrigation schedule" card on Zone Overview (EN+AF keys). **Zone Designer: deliberately zero files/tables/DI touched** (user's explicit caution) — regression-checked in the browser (saved design, results, drag all identical). Browser-verified end-to-end: defaults 120/0.6/50/6 → RAW 36 mm / 6 days / next-due = last-run+6; live recompute (peak 4 → 9 days); save→reload persists; psql shows one upserted row with verbatim values; a temporary 2-day schedule flipped Potato to **Overdue** across list/overview while no-schedule zones kept today's behavior; all 77 dotnet tests green.

- 2026-07-17 — **Zone Designer: hydraulics honesty — editable head pressure, planning caveat, main-flow accumulation.** Three fixes to the results panel's overconfidence: **(1)** the hardcoded 2.0 bar sprinkler-head pressure became an editable Design Input (`DesignInputs.SprinklerHeadPressureBar`, default 2.0 — the property initializer doubles as backward compatibility for persisted design JSON that predates the field); the "Required pump pressure" sub-note now shows the live value. **(2)** A muted caveat line under Results: "Planning estimates — elevation and pipe-fitting losses are not modelled." **(3)** **Main friction now accumulates per segment**: the main splits at each connected lateral's take-off point (new `DesignPipe.DistanceAlongTo` arc-length helper; take-off = the lateral's vertex nearest the main), each segment carries only the flow still to be delivered downstream, and the zero-flow tail past the last take-off contributes nothing.
